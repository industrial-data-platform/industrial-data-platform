# Подключение к KNX демо-стенду

Дата актуализации документа: 2026-03-28

## Назначение

Краткая памятка по текущему подключению к демо-стенду `KNX` для разработки, отладки demo CLI и подготовки первого `KNX`-адаптера в архитектуре `Edge Telemetry Agent`.

## Устройство

- KNX/IP роутер: `Weinzierl KNX IP Router 751`
- Локальный IP роутера задается через `KNX_LOCAL_GATEWAY_IP`
- Порт KNX/IP в локальной сети задается через `KNX_LOCAL_GATEWAY_PORT`
- KNX individual address роутера: `1.1.0`

## Подключение для разработки

Для текущей разработки с этой рабочей станции:

- использовать внешний профиль `external`
- внешний адрес: `${KNX_EXTERNAL_GATEWAY_IP}:${KNX_EXTERNAL_GATEWAY_PORT}`
- NAT-маршрут: `${KNX_EXTERNAL_GATEWAY_IP}:${KNX_EXTERNAL_GATEWAY_PORT} -> ${KNX_LOCAL_GATEWAY_IP}:${KNX_LOCAL_GATEWAY_PORT}`
- именно этот путь сейчас соответствует remote workstation сценарию в архитектуре

Для разработки из локальной сети объекта:

- подключение из локальной сети напрямую к `${KNX_LOCAL_GATEWAY_IP}:${KNX_LOCAL_GATEWAY_PORT}`
- протокол: `KNXnet/IP Tunneling`

Примечание:

- внешний маршрут был проверен и отвечает на `KNXnet/IP`
- при работе из сети объекта предпочтительно использовать локальный адрес `${KNX_LOCAL_GATEWAY_IP}:${KNX_LOCAL_GATEWAY_PORT}`
- при работе с текущей whitelisted workstation дефолтный путь это внешний профиль `external`

## Известные group address

- `0/0/1` = `Switch`
  - командный адрес
  - по этому адресу можно писать команду включения/выключения света
- `0/0/7` = `StatusSwitch`
  - адрес обратной связи по свету
  - ранее в устных обсуждениях мог фигурировать как `SwitchFB`
- `2/0/0` = `Temp Value`
  - аналоговое значение температуры
  - ранее мог называться просто `Temperature`
  - тип со слов заказчика: `2-byte float value`
  - рабочее предположение для KNX: `DPT 9.001` (`°C`)

Предварительное предположение для MVP:

- `0/0/1` и `0/0/7`, вероятно, используют `DPT 1.001`
- `2/0/0`, вероятно, использует `DPT 9.001`
- это нужно окончательно подтвердить по ETS-проекту или таблице адресов

## Правила обработки сигналов

- дискретные сигналы отправляются по изменению значения
- аналоговые сигналы отправляются по достижению настраиваемого порога изменения
- пример для температуры:
- отправлять событие, если изменение относительно последнего отправленного значения составило не менее `1.0 °C`
- периодический опрос включается только для адресов, явно указанных в конфиге
- командные адреса не считаются основным источником телеметрии; для сбора предпочтительны `feedback` и `sensor` адреса

## Роль стенда в текущей архитектуре

- демо-стенд используется как первый `KNX source` для `Edge Telemetry Agent`
- production data path проектируется как `read-only`
- целевой production-сценарий предполагает запуск `Edge Telemetry Agent` в локальной сети объекта рядом с `KNX/IP` роутером
- текущий сценарий разработки для demo-стенда использует удаленный запуск на whitelisted workstation с доступом через `${KNX_EXTERNAL_GATEWAY_IP}:${KNX_EXTERNAL_GATEWAY_PORT}`
- целевой transport для MVP — `MQTT 5.0`; contract-level детали находятся в
  `docs/contracts/edge-telemetry-agent/mqtt-topic-tree.v1.md`
- текущие CLI в репозитории используются как диагностические инструменты и как практическая проверка адресов/режимов подключения

## Что использовать в коде

- для текущей workstation-разработки: профиль `external`
- для запуска из локальной сети объекта: профиль `local`
- transport: `UDP`
- connection mode: `TUNNELING`
- библиотека-кандидат для MVP: `xknx`
- в конфигурации агента использовать `source_id: knx_main`, а адреса стенда задавать как `point_ref`
