# ADR-002: Гибридный режим получения данных `listen + selective read`

Дата: 2026-03-20  
Статус: accepted

## Контекст

В KNX нельзя надежно получить полную картину системы простым активным опросом всех адресов. Часть полезных сигналов приходит только при изменении состояния, часть адресов может не отвечать на отдельный `read`, а без ETS-проекта невозможно корректно трактовать неизвестные group address.

## Решение

Для collector-сервиса принимается гибридный режим:

- основной канал данных: пассивное прослушивание telegrams
- `read_on_start`: только для whitelist `feedback/status/sensor` адресов
- `periodic_read`: выключен по умолчанию и включается только для отдельных адресов
- адреса типа `command` не являются основным источником телеметрии

## Обоснование

- bus monitor дает реальную активность на шине без лишней нагрузки
- selective read позволяет восстановить актуальное состояние после старта или reconnect
- такая схема совместима с неполной картой адресов на этапе MVP

## Последствия

Положительные:

- меньше риск перегрузить шину ненужным polling
- лучше качество стартового состояния по известным адресам
- архитектура остается конфигурируемой и расширяемой

Отрицательные:

- без ETS часть адресов останется только технически наблюдаемой
- требуется аккуратная конфигурация whitelist и правил фильтрации

## Практические правила

- `read_on_start` разрешать только там, где подтверждено, что адрес поддерживает чтение
- `periodic_read` задавать с интервалом не менее `30` секунд, если нет отдельного обоснования
- unknown `DPT` не публиковать как production telemetry
