# ADR-001: Прямая edge-топология и `xknx` как runtime для MVP

Дата: 2026-03-20  
Статус: accepted

## Контекст

Нужно быстро запустить рабочий сбор телеметрии из KNX-стенда с минимальным количеством компонентов. Требуется подключение к KNX/IP роутеру в локальной сети, обработка telegrams, выборочное чтение известных group address и дальнейшая подготовка данных к отправке в веб-контур мониторинга.

Рассматривались варианты:

- прямое подключение из Python через `xknx`
- промежуточный локальный брокер `knxd`
- Java-стек и инструменты `Calimero`

## Решение

Для MVP принимается следующая топология:

- edge-сервис запускается в локальной сети объекта
- сервис подключается напрямую к `KNX/IP` роутеру через `KNXnet/IP Tunneling`
- основной runtime на `Python 3.10+`
- библиотека доступа к KNX: `xknx`
- `Calimero Tools` используются как вспомогательная диагностика, но не как runtime-ядро
- `knxd` не вводится на старте

## Обоснование

- минимальное количество компонентов ускоряет запуск MVP
- `xknx` уже покрывает нужный набор возможностей: callbacks, `TUNNELING`, чтение group address, обработку telegrams
- Python дает быстрый путь к конфигурируемому collector-сервису
- `knxd` полезен, но сейчас добавил бы лишний операционный слой без прямой необходимости

## Последствия

Положительные:

- проще развертывание и меньше точек отказа
- понятный путь от текущих скриптов к сервису collector
- меньше операционной сложности на объекте

Отрицательные:

- если появятся несколько локальных клиентов KNX, может потребоваться отдельный брокер
- часть эксплуатационных возможностей `knxd` пока не используется
- масштабирование на сложные multi-consumer сценарии придется добавлять позже

## Что пересмотреть позже

- ввод `knxd`, если понадобится несколько потребителей KNX-трафика
- переход на `KNX Secure`, если это потребует другой схемы runtime и секретов
