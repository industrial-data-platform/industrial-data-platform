# ADR-007: Хранилища данных `Monitoring & Alarm Platform`

Дата: 2026-05-01
Статус: partially superseded

Примечание 2026-05-04: выбор хранилищ остается принятым. Формулировки про
конкретный Kafka-compatible broker runtime уточнены в `ADR-012`: `Kafka Event
Log` является логическим слоем, локальный MVP использует `Apache Kafka`, а
`Redpanda broker` остается candidate после отдельного compatibility PoC.

Примечание 2026-05-10: термин `Monitoring & Alarm Platform` заменен в текущей
архитектуре на `Industrial Data Platform` плюс отдельные `Web Monitoring Module`
и `Alarm Management Module` (`ADR-014`). Выбор `ClickHouse + PostgreSQL`
остается действующим для data platform core.

## Контекст

`Monitoring & Alarm Platform` должна хранить данные всей платформы, но эти данные
имеют разные профили нагрузки.

- события телеметрии, служебные события, снимки метаданных источников,
  производные события, агрегаты и история тревог являются в основном
  данными, которые только добавляются и не изменяются задним числом
- конфигурации объектов, агентов, источников, точек, правил, политик
  уведомлений, текущий жизненный цикл тревог, подтверждения, приглушения, аудит
  и хранилище IAM являются изменяемым транзакционным состоянием
- целевой промышленный поток уже зафиксирован как
  `MQTT -> Redpanda Connect -> Kafka Event Log -> потребители/аналитика -> хранилища`
- целевой масштаб первого промышленного среза: `100k-1M` точек мониторинга
- целевая свежесть данных в UI/API: секунды, не субсекунды
- развертывание должно поддерживать самостоятельный open-source режим и
  управляемый облачный режим
- один крупный клиент может иметь до `100000` тегов/датчиков
- ожидаемый стартовый профиль: около `10` крупных клиентов с общей схемой
  телеметрии

Из готовых систем и документации видны устойчивые паттерны:

- `ThingsBoard` использует SQL-вариант на `PostgreSQL` или hybrid-подход
  `PostgreSQL + Cassandra/TimescaleDB` для телеметрии и сущностей платформы
- `OpenRemote` хранит постоянные данные в `PostgreSQL`
- `Mainflux` поддерживает компоненты записи в `PostgreSQL`, `TimescaleDB` и
  `MongoDB`
- `Zabbix` использует `PostgreSQL + TimescaleDB` для истории, трендов,
  сжатия и обслуживания старых данных
- `ClickHouse` хорошо ложится на высоконагруженные данные временных рядов,
  которые только добавляются, и событийную аналитику: прием из Kafka, колоночное хранение,
  временное партиционирование, высококардинальные аналитические запросы и
  панели в Grafana
- `Keycloak` требует внешнюю реляционную БД промышленного уровня; `PostgreSQL`
  входит в поддерживаемые промышленные конфигурации

## Решение

Для высоконагруженного промышленного среза принимается разделение хранилищ:

- `Telemetry Store` = `ClickHouse`
- `Platform Store` = `PostgreSQL`

`ClickHouse` является основным аналитическим хранилищем для:

- исходных и канонических событий телеметрии
- служебных событий
- снимков метаданных источников для аналитических соединений
- производных событий
- rollup-таблиц и агрегатов
- истории тревог как неизменяемого событийного слоя

`PostgreSQL` является транзакционным хранилищем платформы для:

- объектов, агентов, источников и реестра точек
- правил тревог и политик уведомлений
- текущего состояния тревог
- подтверждений, приглушений и состояния операторских рабочих процессов
- аудита
- конфигурационного состояния `Platform API`
- базы данных `Keycloak`

`Kafka Event Log` и обслуживающий его Kafka-compatible broker runtime остаются
потоковым, транспортным и слоем повторного чтения событий.
Они не считаются долговременным аналитическим или транзакционным хранилищем
платформы.

## Владение данными и поток данных

- `Telemetry Consumers` читают `Kafka Event Log` и пишут каноническую
  телеметрию, служебные события и снимки метаданных в `ClickHouse`.
- `Streaming Analytics` читает темы Kafka и при необходимости исторический
  контекст из `ClickHouse`, затем пишет агрегаты и производные события в
  `ClickHouse`.
- `Alarm Rule Engine` читает поток телеметрии и производных событий, пишет
  неизменяемую историю тревог в `ClickHouse`, а текущее состояние рабочего
  процесса тревог хранит в `PostgreSQL`.
- `Platform API` читает оба хранилища: `ClickHouse` для истории, графиков и
  аналитики; `PostgreSQL` для конфигурации, правил и текущего состояния.
- `Grafana` читает панели из `ClickHouse`.
- `Keycloak` хранит realm, клиентов и пользователей в `PostgreSQL`.

## Физическая модель телеметрии в ClickHouse

Телеметрия хранится в узкой событийной модели, а не в модели "одна колонка на
тег/датчик".

- одна строка в `telemetry_events` соответствует одному наблюдению одной точки
  в один момент времени
- тег/датчик идентифицируется колонками `tenant_id`, `asset_id`, `source_id`,
  `point_id` и/или `point_key`
- значение хранится в типизированных value-колонках, например `value_float`,
  `value_bool`, `value_string`, `value_type`, `value_raw`
- стабильные метаданные точки хранятся в `PostgreSQL` в реестре точек, а в
  `ClickHouse` попадают как снимки метаданных для исторической аналитики
- добавление нового тега/датчика не должно требовать `ALTER TABLE`

Полная migration-backed физическая модель, contract tables и имена таблиц
фиксируются в `docs/contracts/clickhouse/telemetry-store.v1.md`. Этот ADR
фиксирует решение и принципы модели, а не является source of truth для
ClickHouse schema details.

Правила физической модели:

- `tenant_id` обязателен во всех таблицах `Telemetry Store`
- `event_id` хранится как непрозрачная строка для дедупликации и не ограничивается UUID-only форматом
- `tenant_id` входит в `ORDER BY`, чтобы клиентские панели и API-запросы
  эффективно отсекали чужие данные
- партиционирование выполняется по времени, например `toYYYYMM(ts)`, а не по
  `tenant_id`, `point_id` или `point_key`
- высококардинальные поля не используются как ключ партиционирования, чтобы не
  создавать чрезмерное число частей данных и не ломать фоновые процессы слияния ClickHouse
- для типичных панелей создаются rollup/latest-таблицы, например
  `telemetry_1m_v1`, `telemetry_1h_v1` и `telemetry_latest_v1`
- исходная таблица остается источником истины для повторного чтения,
  расследований и пересчета агрегатов в пределах срока хранения

## Мультитенантность

Для первого промышленного среза принимается модель мультитенантности с общими
таблицами:
все клиенты используют общие таблицы ClickHouse с обязательной колонкой
`tenant_id`.

Это решение выбрано потому что:

- у клиентов одинаковая физическая схема телеметрии
- ожидается около `10` крупных клиентов, а не тысячи клиентов с разными схемами
- модель с общими таблицами проще для миграций, rollup-таблиц и эксплуатации
- ClickHouse рекомендует включать поле клиента в primary key/order key для
  подхода с общими таблицами

Изоляция данных клиентов обеспечивается на нескольких уровнях:

- `tenant_id` присутствует в контексте темы Kafka, каноническом событии и строках
  ClickHouse
- `Platform API` всегда добавляет фильтр клиента из аутентифицированного контекста
- прямой доступ пользователей клиентов к ClickHouse, если он появится,
  ограничивается отдельными пользователями/ролями и политиками строкового
  доступа по `tenant_id`
- эксплуатационные пользователи имеют доступ через отдельные роли и аудит

Отдельная база или отдельные таблицы на клиента не выбираются на старте. Такой
подход можно пересмотреть для клиента с существенно большим объемом данных,
особыми требованиями к изоляции или отдельным управляемым развертыванием.

## Рассмотренные варианты

| Вариант | Оценка |
| --- | --- |
| `PostgreSQL + TimescaleDB` | Хороший путь от `PostgreSQL` и сильная базовая альтернатива для MVP. Отклонен как единственное высоконагруженное решение: для `100k-1M` точек и аналитических панелей `ClickHouse` дает более явное разделение OLAP-нагрузки от транзакционной нагрузки платформы. |
| `ClickHouse + PostgreSQL` | Принят. Разделяет аналитическую нагрузку только на добавление и изменяемое транзакционное состояние, хорошо стыкуется с Kafka-compatible event log, Grafana и высококардинальными запросами. |
| `InfluxDB + PostgreSQL` | Подходит для классической TSDB-телеметрии и поддерживается Grafana. Отклонен из-за отдельной экосистемы запросов/модели данных и менее удобного общего SQL/analytics слоя для событий, соединений и `Platform API`. |
| `Apache IoTDB` | Силен как промышленная IoT-native TSDB с edge-cloud идеологией. Отклонен для первого промышленного среза из-за меньшей распространенности в текущей Python/Kafka/Grafana архитектуре и большего риска внедрения. |
| `Apache Druid / Apache Pinot` | Сильные варианты OLAP в реальном времени для субсекундных панелей и больших потоков. Отклонены как слишком тяжелые для текущего этапа: больше компонентов и операционной сложности, чем нужно для секундной свежести. |
| `Prometheus / VictoriaMetrics` | Хороши для эксплуатационных метрик и Prometheus-compatible monitoring. Отклонены как основное хранилище телеметрии платформы: наша доменная телеметрия включает события, метаданные, историю тревог и API-запросы, а не только сбор метрик и remote write. |
| Одна БД для всего | Отклонено. Высоконагруженная телеметрия, которая только добавляется, и изменяемое состояние платформы имеют разные требования к движку хранения, индексации, срокам хранения, резервному копированию, восстановлению и изоляции запросов. |
| Колонка на каждый тег/датчик в ClickHouse | Отклонено. При `100000` тегов на клиента такая модель создает неуправляемую схему, требует DDL при добавлении точек, усложняет типизацию значений и плохо соответствует запросам вида "точка + интервал времени". |
| Отдельные ClickHouse-таблицы на каждого клиента | Отклонено для первого промышленного среза. Подход может ускорить отдельные запросы для очень крупных клиентов, но усложняет миграции, rollup-таблицы, панели Grafana и эксплуатацию. |

## Последствия

Положительные:

- аналитические запросы, панели и долгую историю можно масштабировать отдельно
  от транзакционного состояния платформы
- `PostgreSQL` остается простым и привычным хранилищем для API, Keycloak,
  конфигурации и состояния рабочих процессов
- `ClickHouse` хорошо соответствует приему через Kafka и крупным таблицам
  временных рядов и событий
- Grafana получает промышленный источник данных для исторических панелей без
  зависимости от live-источника MQTT
- добавление новых тегов/датчиков не требует изменения схемы `ClickHouse`
- мультитенантность явно закреплена через `tenant_id`

Отрицательные:

- появляется два контура постоянного хранения вместо одного
- нужны явные правила владения, резервного копирования, восстановления и
  миграций для обоих хранилищ
- API и аналитические задания должны аккуратно разделять OLTP-запросы и OLAP-запросы
- понадобится контракт схемы таблиц ClickHouse, сроков хранения и
  материализованных представлений
- права доступа должны гарантировать, что фильтр клиента нельзя обойти

## Что пересмотреть позже

- нужен ли отдельный слой озера данных или объектного хранилища для архива за
  пределами срока хранения ClickHouse
- нужен ли управляемый ClickHouse/управляемый PostgreSQL как предпочтительное
  промышленное развертывание
- достаточно ли `PostgreSQL` для текущего состояния тревог при росте
  операторских рабочих процессов
- нужны ли `Druid` или `Pinot`, если появятся требования к массовым
  пользовательским панелям с субсекундной задержкой
- нужно ли вынести отдельного крупного клиента в отдельную таблицу, базу или
  развертывание при существенном перекосе объема данных
